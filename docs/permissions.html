Permissions

How client keys, identities, and access control work in the auth frontend

Identity Model

The Calimero node uses a public-key identity model. Each node has one or more root identities. Applications interact with contexts using client keys — ephemeral keypairs authorized against a root identity:

Root Identity

A permanent keypair held by the node. Used to authorize client keys. Listed via sdk.admin.getIdentities().

Client Key

A short-lived keypair for a specific context. Generated by generateClientKey(). Used to sign RPC calls.

Client Key Generation

// Step 1: get the root identity
const [identity] = await sdk.admin.getIdentities();

// Step 2: generate client key for context + identity
// Note: camelCase in mero-js v2 (was snake_case in v1)
const clientKey = await sdk.admin.generateClientKey({
  contextId: 'ctx_abc',
  contextIdentity: identity.publicKey,
});

// clientKey.publicKey — send to app
// clientKey.privateKey — keep secret, use for signing

What the App Receives

After auth flow completes, the application gets:

access_token
Short-lived JWT. Used as Authorization: Bearer for admin API calls.
refresh_token
Long-lived token for obtaining new access tokens without re-login.
node_url
The Calimero node base URL. Needed to initialize the SDK.
context_id
The context the app should send RPC calls to.
client_public_key
The app's signing identity for this context.
client_private_key
The private half — used to sign RPC requests. Never transmitted to the server.

Security Notes

Private key storage
The client private key should be stored in sessionStorage (cleared on tab close), not localStorage. Never send to any server.
Access token lifespan
Access tokens are short-lived. The auth frontend refresh flow exchanges the refresh token for a new access token automatically.
SSO security
When receiving tokens via URL hash from the desktop app, clear the hash immediately after reading to prevent token leakage in browser history.
